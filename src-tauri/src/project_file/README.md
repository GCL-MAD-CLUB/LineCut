# 项目文件子系统

本目录是项目文件的唯一所有者。软件其余部分只能调用根 `mod.rs` 暴露的无版本接口：规范化路径、读取为当前 `ProjectWorkspace`、将当前 `ProjectWorkspace` 写入文件，以及创建冗余自动备份快照。版本号、文件 DTO、升级过程、明文载荷、密钥和加密协议都不得越过这个边界。

## 分层

- `mod.rs`：极薄的系统边界，只编排读取、逐级升级、当前模型转换和保存。
- `auto_save.rs`：工作区内容哈希、快照命名、变化检测及全局数量轮转。
- `io.rs`：路径、大小限制、同步临时文件和直接替换；不会创建 `.lcp.bak` 文件。
- `handle_v1.rs`：唯一特例。它独占旧容器、V1 bincode DTO、校验和与 V1 解码，借文件名明确提示这不是新协议的一部分。
- `models/`：只存放 V2 及以上的完整冻结 DTO 和相邻版本迁移。
- `protocol.rs`：V2 及以上统一使用的加密信封。内容模型只能看到解密后的载荷，不能自行加密。
- `keyring.rs`：应用密钥选择、密钥 ID 和派生。密钥轮换不能改内容模型或加密信封布局。

## 内容版本不变量

- `models/vN.rs` 发布后冻结，不得跟随软件运行时类型修改。
- 所有 V2+ 模型都必须实现 `ProjectModel`，因此具有相同的版本、解码、编码和 `into_upgrade_parts` 接口。
- `vN.rs` 只通过 `UpgradeFrom<V(N-1)>` 兼容直接上一版，禁止跨版本迁移。
- `models/mod.rs` 是唯一的升级链编排器。导入任何版本都必须逐级得到当前模型，之后才允许转换为软件的 `ProjectWorkspace`。
- 保存永远从当前 `ProjectWorkspace` 创建当前模型。旧项目正常保存后自然成为最新版。

新增 V3 时，在 `models/v3.rs` 复制相同组织并实现 `ProjectModel`、`UpgradeFrom<v2::Model>`；在 `models/mod.rs` 中让 V1 和 V2 均逐级走到 V3，并把 `Current` 切为 V3。不得修改 `handle_v1.rs`、`models/v2.rs` 或既有迁移。

## 固定加密协议

V2+ 使用 XChaCha20-Poly1305 认证加密。每次保存生成独立的 256 位盐和 192 位 nonce，以 HKDF-SHA-256 派生单文件密钥，并把协议版本、内容版本、算法套件、长度、密钥 ID、盐、nonce 和保留位全部作为 AAD 认证。任何密文或已认证文件头修改都会导致打开失败。明文和派生密钥使用 `Zeroizing` 在离开作用域时擦除。

信封预留算法套件、KDF 套件、密钥 ID、flags 和保留区；以后可以轮换应用密钥或增加已预留的实现，而不改变内容版本的组织方式。V1 明文文件只能经 `handle_v1.rs` 导入，保存后统一进入加密协议。

官方构建应提供至少 32 字节且跨版本保持一致的 `LINECUT_PROJECT_BUILD_SECRET_V1`。`build.rs` 在编译期派生并拆分材料，不把该环境变量原文嵌入应用；未提供时只使用供开发构建运行的拆分回退材料。无密码的本地应用必须持有解密能力，因此无法承诺对有能力调试进程、转储内存或逆向二进制的攻击者实现绝对保密；这里提供的是标准强认证加密、密钥分层和提高静态逆向成本，不把混淆冒充成不可破解。

## 自动备份快照

自动备份位于当前缓存根目录的 `LineCut Auto-Save` 子目录，所有项目平铺存放，命名为 `项目名--内容哈希.lcp`。内容哈希只反映当前项目工作区，不包含保存时间、应用版本、随机盐或 nonce；因此定时检测到相同内容时不会重复写入。保留数量对整个自动备份目录生效，写入新快照后删除最旧文件。快照使用与正常项目完全相同的当前内容模型和受保护信封，但不会改变当前项目路径、保存状态或历史游标。
